# Hardware Assignment 1: Structural Multiplexers on Basys 3

This assignment introduced the complete FPGA workflow for COL215: writing VHDL, simulating modules in Vivado, mapping logical ports to Basys 3 board switches/LEDs, synthesizing the design, generating bitstreams, and validating the output on hardware.

The final design builds from primitive gates to reusable multiplexers. Basic `AND`, `OR`, and `NOT` modules are first implemented and tested independently. These primitives are then composed structurally into a 2x1 multiplexer, and the 2x1 multiplexer is reused hierarchically to implement a 4x1 multiplexer.

## Problem Statement

The assignment had two parts:

1. Implement basic logic gates in VHDL and verify them through simulation and board-level execution.
2. Build multiplexers using structural VHDL:
   - A 2x1 multiplexer from `AND`, `OR`, and `NOT` gates.
   - A 4x1 multiplexer from three 2x1 multiplexer blocks.

The implementation was constrained to component instantiation and port mapping rather than behavioral constructs such as `if`, `case`, or process-based mux logic. This made the assignment focus on digital design hierarchy and hardware composition rather than just truth-table encoding.

## Design Approach

### Basic Gate Layer

The primitive gate modules provide the reusable building blocks:

| Module | Function |
| --- | --- |
| `AND_gate.vhd` | Computes the conjunction of two 1-bit inputs. |
| `OR_gate.vhd` | Computes the disjunction of two 1-bit inputs. |
| `NOT_gate.vhd` | Computes the inversion of a 1-bit input. |
| `BASIC_gates.vhd` | Groups the basic gates into one board-testable top-level design. |

This first layer was useful for validating the Vivado setup, simulation flow, constraint file usage, bitstream generation, and Basys 3 programming flow before moving to a larger combinational circuit.

### 2x1 Multiplexer

The 2x1 multiplexer is implemented structurally using two `AND` gates, one `OR` gate, and one `NOT` gate. In the submitted design, the select signal and its complement are used to enable exactly one data input at a time, and the enabled paths are merged through an `OR` gate.

This design choice keeps the hardware mapping transparent: the VHDL hierarchy directly corresponds to the gate-level schematic generated by Vivado.

### 4x1 Multiplexer

The 4x1 multiplexer is implemented recursively using three 2x1 multiplexers:

- Two first-stage 2x1 muxes select between the four data inputs in pairs.
- One final-stage 2x1 mux selects between the two intermediate outputs.

This hierarchy is more modular than writing a flat 4-input Boolean expression. It also demonstrates a reusable hardware-design pattern: validate a smaller block once, then compose it into a larger circuit.

## Design and Hardware Decisions

| Decision | Rationale | Tradeoff |
| --- | --- | --- |
| Structural VHDL instead of behavioral mux logic | Makes the gate-level composition explicit and matches the assignment constraints. | More verbose than a behavioral `with select` or `case` implementation. |
| Separate primitive gate modules | Encourages reusable components and makes simulation/debugging local to each gate. | Adds extra files and component declarations for a small circuit. |
| Hierarchical 4x1 mux from 2x1 muxes | Mirrors how larger muxes are built from smaller mux cells in digital design. | Slightly more structural overhead than a direct Boolean expression. |
| Basys 3 switch/LED mapping | Provides hardware validation beyond waveform simulation. | Requires careful `.xdc` pin assignment and board-specific constraints. |
| Vivado synthesis and utilization reports | Confirms that the structural design is optimized to compact FPGA resources. | Synthesized resource usage may differ from the visible source-level hierarchy because Vivado optimizes logic aggressively. |

## Verification

The assignment was verified through three complementary checks:

1. **Simulation:** VHDL testbenches were used to validate the truth tables for the basic gates, 2x1 mux, and 4x1 mux.
2. **Synthesis and implementation:** Vivado generated schematics and resource utilization reports for the target FPGA.
3. **Board testing:** Bitstreams were deployed on the Basys 3 board, with switches used as inputs and LEDs used as outputs.

## Resource Utilization

Target FPGA: `xc7a35tcpg236-1` on the Basys 3 board.

| Design | LUT Usage | Registers | BRAM | DSP | I/O Buffers |
| --- | ---: | ---: | ---: | ---: | ---: |
| 2x1 mux | 1 LUT3 | 0 | 0 | 0 | 4 |
| 4x1 mux | 1 LUT6 | 0 | 0 | 0 | 7 |

The key observation is that both mux designs synthesize to purely combinational logic with no registers, memory blocks, or DSP usage. Vivado is able to collapse the structural hierarchy into a single LUT for each mux design, which is expected for small Boolean functions on a LUT-based FPGA.

## Reproducing the Design

1. Open Vivado 2022.1 or a compatible version.
2. Create a project targeting the Basys 3 FPGA part `xc7a35tcpg236-1`.
3. Import the relevant VHDL files from the final submission folder.
4. Import the corresponding `.xdc` constraint file.
5. Set the desired mux module as the top-level design.
6. Run behavioral simulation, synthesis, implementation, and bitstream generation.
7. Program the Basys 3 board and test the switch-to-LED behavior.
